solving a very specific business problem: replacing manual email/OneDrive coordination with a controlled document workflow. For an MVP, I'd simplify it even further so you build the smallest system that delivers value.

2. MVP Goal

The application should become a document workflow system, not an ERP.

It only needs to answer three questions:

Which PO?
Which shipment belongs to that PO?
Has customer accepted it?

Nothing more.

16. Biggest Functional Gaps

These are the questions I'd resolve before development:

Gap	Recommendation
Can one PO have multiple invoices?	Yes, model them as Shipments.
Can one invoice have multiple PODs?	Decide now. Prefer one POD per Shipment for MVP.
Can a rejected shipment be replaced?	Yes. Upload a new version and reset approval.
Version history?	Keep previous files instead of overwriting.
Can vendor edit after acceptance?	No. Require a new version to maintain auditability.
Customer visibility	Use a VisibleToCustomer flag instead of deleting or disabling.
File storage	Store files in OneDrive/Blob storage with metadata in the database.
Notifications	Email only for MVP.
Audit trail	Record every upload, approval, rejection, hide/show action with timestamp and user.
Authentication	Role-based login (Vendor, Customer, Admin).
